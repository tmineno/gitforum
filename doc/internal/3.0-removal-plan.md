# v3.0.0 Per-File Removal Plan

Source of truth for Phase 4 removal. Bound by ADR-011.

Classification:

- `KEEP` — used by 3.0 runtime as-is (may be edited; file path stays)
- `KEEP-RENAME` — surface stays, contents heavily rewritten or path changes
- `MOVE-TO-LEGACY` — survives only inside `src/internal/legacy/`. The
  v3.0.0 reachability set is `tests/legacy_gate_test.rs::LEGACY_GATE_PERMANENT_EXEMPTIONS`
  (16 entries: structural + migrate.rs + 11 v2-delegating KEEP files);
  the strict "only the migrate command path" form is the v3.1 target,
  tracked as task `1v400j3l`.
- `DELETE` — removed at Phase 4

Phase reference:

- Phase 0 = preparatory (tasks `lenm78ma`, `xl8jnot9`, `0exewp7g`,
  `t8o3vnt6`, `4w8hm98j`, `hib03vrq`, `3dx6szoh`)
- Phase 1 = additive snapshot path
- Phase 2 = command cutover (CLI surface only — KEEP-file `super::*`
  cleanup deferred to Phase 4 Step 1; see "Phase 4 sweep" below)
- Phase 3 = migration command
- Phase 4 = audit-first removal sweep (task `913c4s9v`):
  Step 0 contamination gate → Step 1 KEEP-file rewiring → Step 2
  MOVE-TO-LEGACY for `event.rs`/`workflow.rs` → Step 3 DELETE-list
  execution → Steps 4-5 main.rs shims and v2 peer types → Step 6
  lock the gate. `internal::legacy/` is **retained** for the migrate
  command per ADR-011 Decision 1.

## New (introduced during Phase 1)

These do not yet exist; listed so the plan covers the full target tree.

Per RFC `7ymtc4b2` node `6af3285f` (proposal 1) confirmed by `85e3720f`:
per-concept codec files (`snapshot/{thread,node,evidence}.rs`) and
`policy/category.rs` are intentionally NOT introduced. SPEC-3.0 defines
one native snapshot schema; the data types and their serde/file-format
helpers live in the existing KEEP files (`thread.rs`, `node.rs`,
`evidence.rs`, `policy.rs`). `snapshot::store` is the sole owner of
tree assembly/disassembly so the SPEC-3.0 §4 schema boundary stays
explicit.

| File | Class | Rationale |
|---|---|---|
| `src/internal/snapshot/mod.rs` | NEW | Snapshot subsystem entry point; re-exports `store` and `link`. |
| `src/internal/snapshot/store.rs` | NEW | Read tip → `ThreadSnapshot`; write `ThreadSnapshot` → tree → commit → CAS. Owns the schema boundary: assembles the tree from per-concept serde structs in `thread.rs`/`node.rs`/`evidence.rs` and validates SPEC-3.0 §4 layout. |
| `src/internal/snapshot/link.rs` | NEW | `links.toml` model + serde. No existing peer model in the v2.0.2 tree, so this lives under `snapshot/` rather than as a peer file. |

`src/internal/legacy/mod.rs` is **not** new — it is the renamed
`src/internal/compat/mod.rs` (see MOVE-TO-LEGACY table above), expanded
during Phase 1 to also re-export `event` and `workflow`.

Deferred (not allocated until evidence demands it):

- `src/internal/snapshot/tree.rs` — split out only when `store.rs` grows a
  second real responsibility or tree-building code needs independent
  tests. Pre-allocating now would re-introduce the file-splitting problem
  proposal 1 is removing.

## Legacy logic inside KEEP files

File-level KEEP means the path persists; it does not mean the contents
are clean. The current `v3.0.0-draft` tip has substantial v1.x/v2.x logic
embedded inside files this plan classifies as KEEP. Every reference must
be removed during Phase 1 or Phase 2; the build-time gate (task
`3dx6szoh`) is what guarantees no residue survives Phase 4.

#### Phase 4 sweep (audit-first)

Tracked as task `913c4s9v`. Driven by RFC `7ymtc4b2` Phase 4 and
ADR-011 Decisions 1 & 3. Earlier revisions of this section described
Phase 4 as a four-step `git rm` sweep that assumed Phase 2 had

`a8afc24` and `251a9fac` on task `913c4s9v` documented that the
cleanup never actually happened, so Phase 4 starts with the
contamination still in place. The sequence below restructures
around that reality.

**Step 0 — contamination audit gate.**
Land `tests/contamination_audit_test.rs` with an `ALLOW_LIST` that

rewire/move/delete the offending file. A `final_audit_pass_*` test
is `#[ignore]`'d until the gate

Companion gate `tests/legacy_gate_test.rs`:

  `LEGACY_GATE_PERMANENT_EXEMPTIONS` (16 entries — structural

### Heuristic for "is this KEEP file done?"

A KEEP file is considered Phase-4-clean when:
1. Its path is **not** in `tests/contamination_audit_test.rs::ALLOW_LIST`
   (the test passes with the file walked normally — i.e. no
   `internal::{event,workflow,timeline,index,reindex,create,write_ops,
   state_change,repair,repair_workflow,prune,purge,github,
   github_import,github_export}` imports remain).
2. It does not reference `EventType`, `DomainEvent`, `Lifecycle`,
   `KIND_PRESETS`, `facet*`, `replay`, `materialize` outside string
   literals or doc comments. (The contamination audit gate catches the
   import side of this; the type/identifier side relies on grep + code
   review until the type relocations in Step 1f-1j land.)
3. The build-time gate from task `3dx6szoh` (`tests/legacy_gate_test.rs`)
   accepts the file. Post-Step-2, that gate's `ALLOW_LIST` no longer
   exempts `event.rs` / `workflow.rs` (they live under `legacy/` so
   their position in the gate flips from importer-allow-list to
   structural-exemption).

`src/main.rs` already meets criterion 1 (current tip — no forbidden
imports remain after the v2.0.2 / Phase 2 main.rs cleanup). Its only
remaining Phase 4 work is the cosmetic + import-path rewire described
in §"Phase 4 sweep" Step 4 above (`preset_lookup` import path moves
when `workflow.rs` relocates to `legacy/`, not deletion).

`src/internal/commands/migrate.rs` is structurally exempt — it is
the migration entry point and the **post-v3.1** sole consumer of
`internal::legacy::*`. **v3.0.0 reality:** it is one of 16 sanctioned
consumers per `tests/legacy_gate_test.rs::LEGACY_GATE_PERMANENT_EXEMPTIONS`,
the others being structural (legacy/{mod,v1,event,workflow}.rs) plus
the 3 v2-delegating 3.0-native modules (`node.rs`, `policy.rs`,
`thread.rs`) and 8 v2 read-path KEEP files (`validate.rs` +
`commands/{doctor,state,thread_new,shared,show,ls,shortlog}.rs`).
v3.1 task `1v400j3l` shrinks the set to the strict 5-entry
structural-plus-migrate baseline.
