# task `913c4s9v`: v3.0.0 Legacy Removal Scope

Status: Accepted (preparatory; ratified together with the v3.0.0 milestone RFC `7ymtc4b2`)
Date: 2026-05-03

## Context

git-forum is migrating from event-chain storage (1.x/2.x) to snapshot-ref
storage (3.0). SPEC-3.0-draft (thread `fg61bcmp`) defines the target model;
CORE-VALUE bounds it. SPEC-3.0 §1.2.4 requires that "3.0 runtime code should
not keep 1.x/2.x event replay semantics alive except in the migration tool,"
and Appendix A enumerates the implementation surfaces that should shrink or
disappear.

RFC `7ymtc4b2` ends with task `913c4s9v` — removal. The risk is
that legacy code lingers in unexpected places (e.g. doctor internals, TUI
rendering, index coherence) and task `913c4s9v` becomes an open-ended excavation
instead of a directory delete.

This ADR fixes the boundary so the deletion is mechanical.

## Decision

1. **Single legacy module tree.** All v1/v2-format-aware Rust code that
   survives v3.0.0 lives under `src/internal/legacy/`. Anything outside
   that tree must compile and run without referencing v1/v2 event types,
   payloads, or projection logic.

2. **Asymmetric import direction.** The build-time gate (preparatory task
   `3dx6szoh`) enforces:
   - `legacy → 3.0` is allowed (migration writes 3.0 snapshots)
   - `3.0 → legacy` is forbidden

3. **Migration-only consumers.** Only the `git forum migrate --to 3.0`
   command path may import from `internal::legacy::*`. The doctor, TUI,
   index, and all read/write commands MUST NOT.

4. **Per-file classification.** Task `913c4s9v` classifies every `src/`
   file as one of:
   - `KEEP` — used by 3.0 runtime as-is
   - `KEEP-RENAME` — surface stays, contents heavily rewritten or path
     changes
   - `MOVE-TO-LEGACY` — survives only inside `internal::legacy/` for
     migration use
   - `DELETE` — removed at task `913c4s9v`

5. **Judgment basis when contested.** When a module's fate is unclear,
   prefer `DELETE`. Re-introduce in v3.1 with measured justification rather
   than carrying speculative legacy. Grounding: CORE-VALUE *Current-State
   Clarity* and *Small Core, Rich Clients*.

6. **Index treated as deletable cache.** Per SPEC-3.0 §9.2, the SQLite
   index is acceleration, not authoritative. v3.0.0 deletes
   `src/internal/index.rs` and `reindex.rs` outright. Search initially
   scans snapshot trees. An index can return in v3.1 if a measured
   bottleneck appears.

7. **GitHub bridge dropped from v3.0.0 core.** Per SPEC-3.0 §13 non-goals,
   the GitHub bridge is not part of 3.0 core. `github*.rs` files are
   `DELETE` for v3.0.0; a separate optional crate may host them later.

## Consequences

Positive:
- task `913c4s9v` becomes a mechanical execution of the DELETE list.
  `internal::legacy/` is retained for the migrate command per Decision 1;
  it is not deleted at v3.0.0.
  Estimated to delete >40% of current `src/internal/*.rs` from
  non-legacy paths.
- The build gate makes accidental reintroduction (a 3.0 module
  importing legacy) fail the build, not just a soft convention.
- The classification table doubles as an onboarding map for any agent
  picking up task `1hg98odf` or task `9635buy0` work mid-stream.

Negative:
- GitHub bridge removal is a user-visible regression for any repo that
  used import/export. Mitigated by SPEC-3.0 §13 already declaring it
  non-core, and by leaving the v2.x branch intact for users who need it.
- Index removal may slow `git forum search` on large repos. Mitigated by
  SPEC-3.0 §9.2 acknowledging tree-scan as acceptable; performance
  return is a v3.1 concern with empirical justification.
- `migrate.rs` essentially rewrites for 3.0; the file is kept by name
  but the implementation is new. Per task `913c4s9v`, the file relocates to
  `src/internal/commands/migrate.rs` during task `1hg98odf` so the application
  command layer is the single home for every CLI subcommand.

## Alternatives considered

**A. Keep the legacy code in `internal::compat/` with a soft "don't import"
convention.** Rejected: convention without enforcement degrades. The
build gate (preparatory task `3dx6szoh`) is the only durable mechanism.

**B. Two-track / feature-flagged dual runtime.** Rejected: directly
contradicts SPEC-3.0 §1.2.4. Doubles the surface for the duration of
tasks `qa8u71j9`, `1hg98odf`, and `9635buy0` and incentivizes carrying both paths into release.

**C. Zero-scratch rewrite.** Rejected: throws away v2.0.2 refactor work
that is already pre-positioned for 3.0 (typed `DomainEvent`,
`internal::commands::*` extraction, `internal::compat::v1` isolation).
Migration tool still requires v1/v2 replay logic, which is exactly what
`compat::v1` is — a fresh repo would re-implement it.

## Exit criteria

1. ADR committed.
2. Task `913c4s9v` covers every existing `src/internal/*.rs`,
   `src/main.rs`, `src/lib.rs`.
3. The 4 sibling preparatory tasks (`lenm78ma`, `4w8hm98j`, `hib03vrq`,
   `3dx6szoh`) reference this ADR.
4. The classification table is the source of truth for task `913c4s9v`
   removal — divergence from the table requires a follow-up ADR.
